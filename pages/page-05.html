Lesson 05

Space vs Time

"Every algorithm consumes two resources: time (operations) and space (memory). Optimising one often costs the other — this is the fundamental trade-off of algorithm design."

Analogy

Two chefs prepare dinner. Chef A has memorised every recipe — ask them anything and they answer instantly, O(1). But their brain holds a finite amount. Chef B reads from a cookbook each time — no memory overhead, but every dish requires a full read, O(n). One trades memory for speed. The other trades speed for memory. Same kitchen, same menu, fundamentally different resource profiles.

The Two Dimensions of Every Algorithm

Time Complexity

Time complexity measures how the number of operations grows as the input size grows. It is what most engineers think of first when they say "Big O." An O(n) time complexity means the algorithm does approximately n operations. An O(n²) means roughly n² operations. Time is the resource that users notice directly — slow algorithms mean slow apps.

Space Complexity

Space complexity measures how much memory the algorithm allocates as the input size grows. Crucially, we only count the extra memory — not the input itself. If you pass in an array of 1,000 elements, that memory belongs to the caller, not the algorithm. Space complexity counts what your algorithm additionally creates: new arrays, hash maps, recursive call stack frames, auxiliary variables.

Key Insight — Auxiliary Space

When we say an algorithm has O(1) space complexity, we mean it uses a constant amount of extra memory — a few variables — regardless of input size. The input array itself is not counted. This distinction is called auxiliary space, and it is what space complexity always refers to.

The Trade-off in Practice

The most classic trade-off is memoisation: storing the result of expensive function calls so you never recompute them. A recursive Fibonacci function is O(2ⁿ) time and O(n) space. Add a cache (a hash map of previous results) and it becomes O(n) time and O(n) space. You spent memory to buy time — and the deal was worth it by several orders of magnitude.

The reverse trade-off appears in streaming algorithms. Processing a file too large to fit in memory means you must handle one chunk at a time — O(1) or O(k) space — but you cannot take shortcuts that require seeing the full dataset, so time suffers. Choosing which direction to optimise is a judgment call based on your system constraints.

When Memory Is the Bottleneck

On mobile devices, embedded systems, or services with strict memory limits, space complexity is not secondary to time complexity — it is often the primary constraint. A function that builds an O(n) lookup table might be the right answer on a server with 32GB RAM, and completely unacceptable on a device with 128MB. Good engineers always ask: what is the memory budget for this context?

Pseudocode — Same Problem, Two Trade-offs

Pseudocode
// APPROACH 1: Save memory, sacrifice time — O(n²) time, O(1) space FUNCTION hasDuplicateSlow(arr): FOR i FROM 0 TO len(arr): FOR j FROM i+1 TO len(arr): IF arr[i] == arr[j]: RETURN true RETURN false // No extra memory, but checks every pair → O(n²) // APPROACH 2: Spend memory, buy time — O(n) time, O(n) space FUNCTION hasDuplicateFast(arr): seen ← empty set // O(n) space — the trade-off cost FOR EACH item in arr: IF seen contains item: RETURN true // O(1) lookup every time ADD item to seen RETURN false // One pass, O(1) lookup → O(n) time

Implementation — hasDuplicate Two Ways

JavaScript 
// ─────────────────────────────────────────────────────────────
// VERSION 1 — O(n²) time | O(1) space
// No extra memory. Compares every pair. Slow at scale.
// ─────────────────────────────────────────────────────────────
function hasDuplicateSlow(arr) {
  for (let i = 0; i < arr.length; i++) {       // n iterations
    for (let j = i + 1; j < arr.length; j++) { // n-i iterations each
      if (arr[i] === arr[j]) return true;      // found a pair
    }
  }
  return false;
  // Space: O(1) — only i and j variables
  // Time:  O(n²) — n(n-1)/2 pairs checked
}

// ─────────────────────────────────────────────────────────────
// VERSION 2 — O(n) time | O(n) space
// Trade memory for speed. One pass. Fast at scale.
// ─────────────────────────────────────────────────────────────
function hasDuplicateFast(arr) {
  const seen = new Set();  // O(n) space — grows with input

  for (const item of arr) {
    if (seen.has(item)) return true; // O(1) lookup — hash set
    seen.add(item);
  }
  return false;
  // Space: O(n) — set stores up to n elements
  // Time:  O(n) — single pass, O(1) per lookup
}

TypeScript 
// ─────────────────────────────────────────────────────────────
// VERSION 1 — O(n²) time | O(1) space
// ─────────────────────────────────────────────────────────────
function hasDuplicateSlow<T>(arr: T[]): boolean {
  for (let i: number = 0; i < arr.length; i++) {
    for (let j: number = i + 1; j < arr.length; j++) {
      if (arr[i] === arr[j]) return true;
    }
  }
  return false;
}

// ─────────────────────────────────────────────────────────────
// VERSION 2 — O(n) time | O(n) space
// ─────────────────────────────────────────────────────────────
function hasDuplicateFast<T>(arr: T[]): boolean {
  const seen: Set<T> = new Set<T>();

  for (const item of arr) {
    if (seen.has(item)) return true;
    seen.add(item);
  }
  return false;
}

Common Trade-off Patterns

PatternTimeSpaceWhen to use
Brute force O(n²) O(1) Tiny datasets, memory is severely constrained
Hash set / map cache O(n) O(n) General case — trading memory for speed is almost always worth it
Sorting + two pointers O(n log n) O(1) When sort cost is acceptable and memory is limited
Memoisation (top-down DP) O(n) O(n) Recursive problems with overlapping subproblems
Tabulation (bottom-up DP) O(n) O(1)–O(n) When you can reduce the table to a rolling window

Where You'll See This in the Real World

Redis Caching

Redis sits entirely in RAM — the ultimate space-for-time trade. A database query that takes 80ms becomes a 0.2ms cache hit. The cost: memory is expensive.

Browser localStorage

Browsers cache computed results (CSS layout, compiled JS) in localStorage to avoid recomputing. The trade: startup cost drops, disk usage rises.

CDN Edge Caching

CDNs store copies of content at edge nodes worldwide — massive space cost — so every user request is O(1) local retrieval instead of a cross-continental round trip.

Mobile Memory Limits

iOS terminates apps that consume too much RAM. Mobile algorithms often prefer O(n log n) time with O(1) space over O(n) time with O(n) space — the trade-off flips.
