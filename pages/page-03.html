Complexity Analysis

"Complexity analysis measures algorithm efficiency in terms of operations relative to input size — not in seconds or milliseconds."

Analogy

Two runners race a marathon. Runner A has newer shoes and runs on a faster track — but uses a terrible route. Runner B has older gear — but took a masterclass in optimal routing. At 5km, A is ahead. At 42km, B wins by an hour. A faster machine running a slow algorithm always loses to a slower machine running a brilliant one — at scale.

How Complexity Analysis Works

Why Not Just Measure in Seconds?

Timing an algorithm in seconds is machine-dependent. Your laptop runs the same function three times faster than a Raspberry Pi and ten times slower than a data-centre server. The measurement changes every time you upgrade your hardware, switch languages, or run the code under different system load. It tells you nothing portable or predictable.

Complexity analysis instead counts abstract operations — comparisons, assignments, index accesses — and expresses how that count grows relative to the input size. This number is machine-independent, language-independent, and holds true forever.

What "n" Means

The variable n represents the size of the input your algorithm operates on. If you pass an array of 100 elements, n = 100. If you pass 1,000,000 elements, n = 1,000,000. The goal of complexity analysis is to describe the relationship between n and the number of operations your algorithm performs. Does doubling n double the work? Square it? Leave it unchanged? That relationship is the complexity.

Key Insight

n is not always an array length. For a string, n is the number of characters. For a graph, n might be the number of nodes or edges. For a matrix, n might be rows × columns. Always ask: what is the primary input dimension this algorithm scales with?

Best, Worst, and Average Case

Any given algorithm can behave differently depending on the structure of its input. Consider searching for a value in an unsorted array. If the target is the very first element, you find it in one step — that is the best case. If it is the last element or not present at all, you check every element — that is the worst case. The average case assumes random distribution and calculates the expected number of steps across all possible inputs.

Best Case — Ω (Omega)

The minimum number of operations under the most favourable input. Rarely the focus — best case is almost never guaranteed in production.

Worst Case — O (Big O)

The maximum operations under the most unfavourable input. This is what we design and optimise for. If the worst case is acceptable, everything else is.

Average Case — Θ (Theta)

Expected operations averaged across all valid inputs. Useful for probabilistic analysis but harder to calculate. Quicksort is O(n²) worst but O(n log n) average.

Why We Always Design for the Worst Case

In systems that serve real users, you do not get to choose which inputs arrive. A malicious actor, an unexpected data shape, or simply a large dataset can all trigger worst-case behaviour at any moment. Designing for average case is designing for a world where everything goes well — which is not the world software runs in.

When a senior engineer says "this function is O(n)," they mean its worst-case behaviour is linear. That guarantee is what lets you reason safely about how a system will behave under load.

Pseudocode — Linear Search with Operation Counts

Pseudocode
FUNCTION linearSearch(arr, target): i ← 0 // 1 operation — assignment WHILE i < arr.length: // n+1 comparisons (loop condition) IF arr[i] == target: // n comparisons (worst case) RETURN i // 1 operation — best case exit i ← i + 1 // n operations — increment RETURN -1 // 1 operation — not found // Total: 3n + 3 operations in worst case // Drop constants and coefficients → O(n) // Space: O(1) — no extra data structures used

Implementation

JavaScript 
/**
 * linearSearch — Find the index of target in arr.
 *
 * Complexity breakdown:
 *   - Initialise i:       1 op
 *   - Loop condition:     n+1 comparisons
 *   - Element comparison: n comparisons (worst case)
 *   - Increment i:        n additions
 *   Total: 3n + 2 → O(n) time, O(1) space
 */
function linearSearch(arr, target) {
  for (let i = 0; i < arr.length; i++) { // n iterations worst case
    if (arr[i] === target) {              // O(1) per comparison
      return i;                           // best case: O(1) if target is arr[0]
    }
  }
  return -1;                              // not found
}

// ── Cases ───────────────────────────────────────────────
const data = [4, 11, 2, 88, 7, 3, 42];

// Best case  — O(1): target is at index 0
console.log(linearSearch(data, 4));   // 0  — 1 comparison

// Worst case — O(n): target is last or absent
console.log(linearSearch(data, 42));  // 6  — 7 comparisons
console.log(linearSearch(data, 99));  // -1 — 7 comparisons

// We say the function is O(n) because Big O
// always describes the WORST case.
TypeScript 
/**
 * linearSearch — Find the index of target in arr.
 * Time: O(n) | Space: O(1)
 */
function linearSearch<T>(arr: T[], target: T): number {
  for (let i: number = 0; i < arr.length; i++) {
    if (arr[i] === target) return i;
  }
  return -1;
}

// Generic version works on any comparable type
const nums: number[] = [4, 11, 2, 88, 7];
const words: string[] = ['apple', 'banana', 'cherry'];

console.log(linearSearch(nums,  88));        // 3
console.log(linearSearch(words, 'banana')); // 1

// TypeScript enforces type consistency — prevents:
// linearSearch(nums, 'hello'); // Error: string not assignable to number

Where You'll See This in the Real World

Database Query Planning

PostgreSQL's query planner performs complexity analysis on every query to decide whether to use a sequential scan O(n) or index scan O(log n).

React Render Optimisation

React's reconciliation algorithm is O(n) by design — it compares the virtual DOM in a single pass. Earlier approaches were O(n³) and made large UIs unusable.

API Rate Limiting

A sliding-window rate limiter must check request counts in O(1) or O(log n). An O(n) check per request would itself become the bottleneck under load.

Compiler Analysis

Compilers analyse the complexity of loops and inline or unroll code based on whether the loop is O(1) (constant iterations) or O(n) — automatic optimisation at build time.
