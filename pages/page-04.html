Lesson 04

Big O Notation

"Big O notation is the universal language engineers use to describe how an algorithm's resource usage grows as input size grows."

Analogy

Imagine searching for a book in a library. O(1) — you know the exact shelf and slot, you walk straight to it. O(log n) — you use the card catalogue: open to the middle, go left or right, halve again, find it in a few steps. O(n) — the catalogue is broken, you walk every aisle checking each spine. O(n²) — you compare every book against every other book to find duplicates. Same library, wildly different costs.

Understanding Big O

What Does the "O" Stand For?

The "O" stands for Ordnung, the German word for "order of magnitude." It was introduced by the mathematician Paul Bachmann in 1894. Big O notation gives us the order of growth of a function — not its exact value, but the shape of how it scales. O(n²) does not mean exactly n² operations. It means that as n grows, the operations grow proportionally to n².

How to Read O(n²) Aloud

Say: "Order n squared" or "Big O of n squared". Never "O n 2" or "oh en squared". The phrasing matters in interviews and team discussions because it signals you understand it as a growth relationship, not a formula. Similarly: O(log n) is "order log n", O(1) is "order 1" or "constant time".

The Three Rules of Big O

Rule 1 — Drop Constants

O(2n) simplifies to O(n). O(500) simplifies to O(1). Constants are irrelevant because we care about growth shape, not exact counts. A constant factor never changes the order of magnitude.

Rule 2 — Drop Non-Dominant Terms

O(n² + n) simplifies to O(n²). At large n, the n term becomes negligible next to n². Keep only the term that grows fastest — it will dominate all others.

Rule 3 — Different Inputs, Different Variables

A function that loops through array A then array B is O(a + b), not O(n). Use different variable names for different inputs. Only combine them if the arrays are the same input.

How to Calculate Big O by Counting Loops

The fastest way to estimate complexity is to count your loops. A single loop over n elements is O(n). A loop inside a loop, both running over n elements, is O(n²). Three nested loops: O(n³). A loop that halves the problem each iteration (like binary search) is O(log n). Each loop you add multiplies the complexity; each halving adds a log factor.

Pro Tip

When you see code that loops n times inside a function that is called n times, the total is O(n²) — even if the outer repetition is not an explicit loop. Watch for recursive calls, map/filter chains, and nested array methods hiding quadratic complexity.

Pseudocode — O(n) vs O(n²)

Pseudocode
// ── O(n) — one loop ───────────────────────────── FUNCTION printAll(list): FOR EACH item in list: // runs n times PRINT item // O(1) each → total O(n) // ── O(n²) — loop inside a loop ────────────────── FUNCTION printPairs(list): FOR EACH a in list: // runs n times FOR EACH b in list: // runs n times for each a PRINT a, b // O(1) each → total n × n = O(n²) // With n=1,000: printAll → 1,000 ops // printPairs → 1,000,000 ops

Implementation — Four Complexity Classes

JavaScript 
// ── O(1) — Constant Time ──────────────────────────────────
// Does exactly the same number of operations regardless of n.
function getFirst(arr) {
  return arr[0];  // 1 operation always — index access is O(1)
}
getFirst([1, 2, 3]);              // 1 op
getFirst(new Array(1_000_000)); // still 1 op

// ── O(n) — Linear Time ───────────────────────────────────
// One operation per element. Doubles when n doubles.
function sumAll(arr) {
  let total = 0;
  for (let i = 0; i < arr.length; i++) { // loops n times
    total += arr[i];                        // O(1) per iteration
  }
  return total;
}

// ── O(n²) — Quadratic Time ───────────────────────────────
// Every element compared to every other element.
// Watch for this pattern — it is the most common bug at scale.
function printPairs(arr) {
  for (let i = 0; i < arr.length; i++) {      // n iterations
    for (let j = 0; j < arr.length; j++) {    // n iterations each
      console.log(arr[i], arr[j]);            // n × n = n² total
    }
  }
}

// ── O(log n) — Logarithmic Time ──────────────────────────
// Halves the search space each iteration.
// Requires a SORTED array. n=1,000,000 → max 20 comparisons.
function binarySearch(arr, target) {
  let left  = 0;
  let right = arr.length - 1;

  while (left <= right) {
    const mid = Math.floor((left + right) / 2); // find midpoint

    if      (arr[mid] === target) return mid;  // found it
    else if (arr[mid] <  target) left  = mid + 1; // target is right half
    else                           right = mid - 1; // target is left half
    // Each iteration: search space halves → log₂(n) steps total
  }
  return -1;
}

// ── O(1) — Constant Time ──────────────────────────────────
function getFirst<T>(arr: T[]): T | undefined {
  return arr[0]; // 1 operation always
}

  let total: number = 0;
  for (let i: number = 0; i < arr.length; i++) {

  }
  return total;
}

Where You'll See This in the Real World

Search Engines

Google's inverted index enables O(1) keyword lookup across trillions of pages. Without it, every search would be O(n) over the entire web — physically impossible.

Large Dataset Sorting

JavaScript's Array.sort() uses TimSort — an O(n log n) hybrid algorithm. Choosing bubble sort instead for 1M records would be 150,000x slower.

Network Routing

BGP routing tables use prefix tries for O(log n) route lookups. Routers handle millions of packets per second — O(n) lookup per packet would collapse the internet.

DNA / Genomics Matching

Sequence alignment algorithms like Smith-Waterman are O(mn). Researchers optimize aggressively — naive O(n²) on a 3-billion-base genome would take centuries.
